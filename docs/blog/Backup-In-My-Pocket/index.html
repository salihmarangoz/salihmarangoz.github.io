Efficient Personal Backup Strategies using BTRFS

My differential backup strategy using rsync and an external storage formatted with BTRFS+LUKS which supports encryption, on-the-fly compression, de-duplication, and filesystem-level snapshot.

Published on July 13, 2022


1. Introduction

I am introducing my backup strategy using a USB flash drive attached to my keys. You may adapt it to SSD, HDD, and other storage types. With strategy you will be able to:

  • Have a backup flash drive attached to your keys.
  • Compress and de-duplicate backup contents for the maximum storage space efficiency.
  • Encrypt data for privacy in case of losing the backup medium.
  • Do a differential backup quickly. (Incremental backup is also possible.)

To achieve these needs I have decided on using BTRFS+LUKS because it supports

  • Volume encryption.
  • Transparent on-the-fly compression (ZLO, ZSTD, etc.)
  • Transparent de-duplication (offline)
  • Filesystem-level snapshot
  • Performance improvements for SSD (e.g. TRIM support, life improving tricks) and HDD (e.g. data locality optimizations).
  • Efficient storage for small files. BTRFS stores very small files in the metadata instead of wasting storage blocks.

This strategy also have some disadvantages:

  • De-duplication increases fragmentation. De-fragmentation may distrupt de-duplication. Not both at the same time.
  • Accessing files from Windows is possible. But with LUKS, things may not go well.

2. Prerequisites

  • >= Ubuntu 18.04
  • Bash terminal
  • An external storage device (will be formatted). Find a good storage drive for your backups. Things you can keep in mind are:
    • Storage size: Think about all the files you need to backup and the future. You will be able to exclude some folders.
    • Strong chassis: Select a storage with a tough (e.g. metal) key-chain hole.
    • Speed: You will not need something very fast since only the difference will be transferred.
    • My choice: Samsung USB 3.1 Flash Drive BAR Plus 256GB

3. Setup

3.1. File System

In the section I have used my old flash drive for demonstration since I don’t want to wipe my backup medium.

These steps will wipe your storage device and format to BTRFS+LUKS. Make sure there is no important data inside. Firstly, we need btrfs-progs to be able to use BTRFS:

$ sudo apt install btrfs-progs

Open Disks with Ubuntu Search. Click to 3-dots and format your backup disk. If some partitions are mounted try unmounting them first. I would recommend GPT partition table since it supports partitions with >2TB.

Before creating our main partition, I would recommend creating a 64MB FAT32 partition so that you can put some files inside and other people can open it on Windows. For the main part, select “Other”…

Set a meaningful volume name and copy/note it. Also choose “other” and continue…

Select “Btrfs”. Also select “LUKS” if you would like to encrypt the contents, especially for flash drives attached to keychain.

Set your password for LUKS. If you can forget the password make sure to note it down somewhere. It may be impossible to unlock it without the password.

After creating the main volume, select BTRFS partition (not the LUKS partition) and copy/note UUID.

$ sudo gedit /etc/fstab

Add the following line at the end of /etc/fstab and don’t forget to update UUID and mount target. My flash drive’s UUID was da186ce0-a849-4055-98ec-9f36e56fc01c and volume name was MyUsbBackup. If your CPU is not good you can use compress=zstd:5. In BTRFS, compress tries to compress a small portion of the content, if it can be compressed well then tries to compress all of it. compress-force tries to compress all the content without any heuristic approaches. I would recommend using compress-force for backup medium.

/dev/disk/by-uuid/da186ce0-a849-4055-98ec-9f36e56fc01c /mnt/MyUsbBackup auto nosuid,nodev,nofail,x-gvfs-show,noatime,compress-force=zstd:15,noauto 0 0

Reload fstab with this command:

$ sudo mount -a

Plug out and plug in your flash drive and check if it is mounted correctly:

$ df -h

Output on my system:

/dev/dm-0                                   3,8G  3,6M  3,3G   1% /mnt/MyUsbBackup

If you can see it is mounted under /mnt folder then everything is alright.

3.2. Script

For synchronizing and de-duplication we need these two packages:

$ sudo apt install rsync duperemove

Copy paste this script at the end of ~/.bashrc so that you can call this function from all terminals.

function backup_to_MyUsbBackup {
    BACKUP_SOURCE_DIR="$HOME/MyFiles"
    BACKUP_DESTINATION_DIR="/mnt/MyUsbBackup"
    TEST_DEST_FILE="$BACKUP_DESTINATION_DIR/.BACKUP_DESTINATION_MyUsbBackup"
    TEST_SRC_FILE="$BACKUP_SOURCE_DIR/.BACKUP_SOURCE_MyUsbBackup"
    
    if test -f "$TEST_SRC_FILE"; then
        if test -f "$TEST_DEST_FILE"; then
            rsync --dry-run -aP --delete --exclude-from "$BACKUP_SOURCE_DIR/.rsync_exclude.list" "$BACKUP_SOURCE_DIR" "$BACKUP_DESTINATION_DIR"
            duperemove -dr --hashfile="$BACKUP_SOURCE_DIR/.dedup.hash" "$BACKUP_DESTINATION_DIR"
        else
            echo "$TEST_DEST_FILE not found!"
        fi
    else
        echo "$TEST_SRC_FILE not found!"
    fi
}

Create empty .BACKUP_DESTINATION_MyUsbBackup file at the destination (backup storage) and .BACKUP_SOURCE_MyUsbBackup at the source (folder to backup). In your MyFiles directory create a file named MyUsbBackup.list for file/folder exclusions and add test files listed below. If you don’t add these files, they will be removed. Also, add files/folders you would like to exclude from rsync to this file:

.BACKUP_DESTINATION_MyUsbBackup
.BACKUP_SOURCE_MyUsbBackup
.dedup.hash

Then open a new terminal and run command backup_to_MyUsbBackup. I put some files as an example and the output on my terminal looks like this:

No dedupe candidates found.

If everything looks perfect then delete --dry-run in the script to actually start copying and/or removing files. Note that synchronization means copying some files and deleting some others. If you want to disable deleting files then please remove --delete in the script.

Now let’s check the efficiency of this backup strategy:

$ sudo compsize /mnt/MyUsbBackup

It can be seen that total contents in MyFiles was 834M. De-duplication detected that Tide Prediction.csv and Tide Prediction (copy).csv are the same files so used space in the flash drive decreased to 417M. After compression (since csv files can be easily compressed) disk usage decreased to 39M. In total, 795M disk space is saved. Note that, the compression performance will be lower with already compressed files such as mp4, mp3, jpg, etc.

4. An Important Warning

Please don’t plug out your external storage devices without un-mounting first. Linux buffers filesystem operations and files you are seeing on file explorer may not be written to the storage device yet. Sometimes un-mounting takes more than 10 minutes depending on your computer and external storage write speeds, so please be patient. Otherwise, your partition may get corrupted.

5. Results

I would like to share my results with this strategy. I am using the same backup strategy for my 256G flash drive and 1T hard disk. I find the results below pretty good. These results may be different for you depending on your habits and works.

For my 256G flash drive; 232G Original -> 206G De-duplicated -> 196G Compressed. 36G saved in total!

none       100%      177G         177G         197G

For my 1T external drive; 764G Original -> 606G De-duplicated -> 558G Compressed. 206G saved!

Note: On the external drive, I was using zlib with compress, but later on changed to zstd with compress-force. I would save a few more space if I were to do it from the beginning.

Hash files for offline de-duplication are using 563,8M for my external drive, and 162,6M for my flash drive. Of course, I have excluded these files from rsync. Moreover, storing hashes speed up the de-duplication process a lot, for example it takes 5 seconds for my flash drive to check de-duplication for my flash drive (when filesystem metadata is cached). I have also tried not storing hashes but computing all hashes for all the files was too time consuming.

In terms of speed, my weekly backup to my flash drive and external drive usually takes 5-10 minutes including rsync+compression+de-duplication.

In the future, I am planning to buy larger storage devices (>4TB) and using snapshot feature of BTRFS to have incremental backups. It is still possible to use this feature but accidentally adding huge files into snapshots and removing them is a bit hassle for me. Also, I am planning to use BTRFS in my computer’s root partition so that I can take snapshots before upgrading installed packages.
